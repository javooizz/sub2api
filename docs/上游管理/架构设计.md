# 架构设计

## 一、数据模型

migration:`backend/migrations/149_upstream_provider_management.sql`(建表)+ `150_upstream_drop_proxy_id.sql`(移除 per-provider `proxy_id`,代理改全局)+ **`151_upstream_usage_collection.sql`(消耗采集:`recharge_ratio` 列 + `upstream_usage_daily` + `upstream_usage_cursor`)**。启动时 embed FS 自动应用、checksum 校验、幂等。ent schema 真相源:`backend/ent/schema/upstream_provider.go`、`upstream_change_event.go`、`notify_channel.go`、`upstream_usage_daily.go`、`upstream_usage_cursor.go`。

> ⚠️ **新增 ent 实体必须配套手写 SQL 迁移**:本仓库运行期建表靠 `migrations/*.sql`(非 ent 自动建表),`make generate` 只生成 ORM 代码、**不**生成迁移。漏写迁移会导致表在运行时不存在(见 `实施验证与已知问题.md` §六)。

### 1. `upstream_providers` — 上游站点

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | varchar(100) | 名称 |
| `type` | varchar(20) | `sub2api` \| `newapi` |
| `site_url` | varchar(500) | 官网地址,登录与数据采集;允许含路径前缀(反代) |
| `api_base_url` | varchar(500) | 模型网关地址,空=同 site_url;仅用于关联帐号匹配与 Token 展示 |
| `status` | varchar(30) | `active` \| `credential_error` \| `unreachable` \| `disabled` |
| `credentials` | jsonb | **Sensitive**:username/password/access_token/user_id/cf_clearance/... |
| `balance_threshold` | float8? | 余额告警阈值,空=不告警 |
| `notify_on_price_change` | bool | 价格/模型变动是否通知 |
| `refresh_interval_minutes` | int | 刷新间隔,默认 60(也作消耗采集节奏) |
| `recharge_ratio` | float8 | **充值比例 N(1:N,¥1 充得 $N 额度)**,默认 1;真实成本¥ = cost_usd ÷ N |
| `latest_snapshot` | jsonb | 最近一次归一化快照 |
| `last_refreshed_at` | timestamptz? | |
| `last_error` | text | 最近错误(已脱敏,不含凭证/响应原文) |
| `consecutive_failures` | int | 连续失败计数 |
| `balance_alerted` | bool | 余额已告警标记(跨越式去重) |
| `refresh_started_at` | timestamptz? | **刷新抢占锁**:非空=刷新进行中,5 分钟 stale 可重抢 |
| `remark` | text | 备注 |

索引:`status`。

### 2. `upstream_change_events` — 变更事件(只追加)

`provider_id` / `type`(40) / `summary`(text) / `detail`(jsonb) / `notified`(bool)。
索引:`(provider_id, created_at DESC)` 供游标分页。
事件类型:`balance_low` / `balance_recovered` / `price_changed` / `model_added` / `model_removed` / `group_added` / `group_removed` / `refresh_failed` / `credential_error`。

> 删除上游时由 repository 在**事务内级联删除**事件 + service 层清理诊断截图,不依赖 DB 级联。

### 3. `notify_channels` — 通知渠道(通用,scope 隔离)

`name` / `type`(email\|webhook) / `scope`(当前固定 `upstream`) / `enabled` / `events`(jsonb,订阅事件类型,空=全部) / `config`(jsonb) / `last_sent_at` / `last_error`。
索引:`scope`。

### 4. `upstream_usage_daily` — 消耗按日 × 维度 rollup(fork 新增)

| 字段 | 类型 | 说明 |
|------|------|------|
| `provider_id` | bigint | 上游 id |
| `day` | timestamptz | 配置时区当天 00:00(关账粒度) |
| `scope_type` | varchar(16) | `total` \| `key` \| `group` \| `model` |
| `scope_key` | varchar(128) | **稳定身份键**:密钥=token_id/api_key_id;分组=名;total="" |
| `scope_name` | varchar(200) | 最新显示名快照(改名只动此列) |
| `requests` | int | 请求数 |
| `tokens` | bigint | token 数 |
| `cost_usd` | decimal(20,10) | 上游实扣(USD);¥ 读时按 `recharge_ratio` 换算,**不落库** |

索引:唯一 `(provider_id, day, scope_type, scope_key)`、查询 `(provider_id, scope_type, day)`。

### 5. `upstream_usage_cursor` — 采集游标 + 乐观租约(每 provider 一行,fork 新增)

`provider_id`(唯一)/ `collect_started_at`(**采集租约**,非空=进行中,stale 10min 可重抢,与刷新锁隔离)/ `collected_through_day`(增量水位)/ `backfill_done` / `backfill_oldest_day` / `last_collected_at` / `last_error` / `last_partial` / `partial_reason`(usage 采集状态的**唯一可靠来源**,与快照 `Partial` 无关)。

> 删除上游时由 repository 在**同一事务内级联删除** `upstream_usage_daily` + `upstream_usage_cursor`(同事件,不依赖 DB 级联)。

## 二、采集适配器

### 统一快照 `domain.UpstreamSnapshot`

两种上游归一化为同一结构:`Balance *float64` / `Currency` / `UserInfo` / `Groups[]{Name,Ratio,Description,Models}` / `ModelPricing map[string]json.RawMessage` / `Partial bool`。

> **Partial 语义**:任一子接口失败时对应字段缺失且 `Partial=true`,**不整体报错**。diff 时缺失的部分跳过对应比较,避免误报"全部分组被移除"。

### new-api 适配器(`upstream_adapter_newapi.go`)

| 接口 | 用途 | 失败处理 |
|------|------|----------|
| `GET /api/status` | `quota_per_unit`(余额换算分母) | 回退 500000 + 标 partial |
| `GET /api/user/self` | 余额(`quota ÷ quota_per_unit`)、用户 | **认证失败上抛**(续期依赖) |
| `GET /api/user/models` | 用户可用模型 | partial |
| `GET /api/pricing` | 模型价格 + `enable_groups` + `group_ratio` | partial |
| `GET /api/log/self`(`FetchUsageDaily`) | **消耗采集**:翻页拉日志,`type∈{2 消费,6 退款}` 计净额,按 `token_id`/`group`/`model`/`total` 归集 | partial |

认证头:`Authorization: Bearer <access_token>` + `New-Api-User: <user_id>`。
分组模型口径:用户可用模型 ∩ 分组 `enable_groups`(用户模型列表缺失时不做交集过滤)。
账密续期(`errUpstreamAuth` 触发):`POST /api/user/login` 拿 session → `GET /api/user/token` 换系统 access token;**该 token 接口即便已有 session 也要求 `New-Api-User: <userID>` 头**(部分 fork 缺失即 401),已随登录返回的用户 ID 一并发送。

### sub2api 适配器(`upstream_adapter_sub2api.go`)

对接的是**本软件自身**的 API(响应形如 `{code:0,message,data}`):

| 接口 | 用途 |
|------|------|
| `POST /api/v1/auth/login` `{email,password}` | 登录;成功 `data.access_token`,2FA 时 `data.requires_2fa` → 拒绝 |
| `GET /api/v1/user/profile` | `{id,username,balance}` |
| `GET /api/v1/groups/available` | `[{name,rate_multiplier,description}]` |
| `GET /api/v1/model-plaza` | `{enabled, models:[{name,groups:[{name}]}]}` 倒排成分组→模型;`enabled=false`→partial |
| `GET /api/v1/keys` / `POST /api/v1/keys` | Token 列表 / 创建(明文 key) |
| `GET /api/v1/user/api-keys/:id/usage/daily`(`FetchUsageDaily`) | **消耗采集**:逐密钥按日(`date`/`requests`/`total_tokens`/`actual_cost`),汇 key + total;**无分组维度** |

认证头:`Authorization: Bearer <access_token>`。JWT 的 `exp` 解析为 `token_expires_at`(不验签)。

### 共用 HTTP 层(`upstream_http.go`)

每 provider 构造:代理(「采集设置」全局 `upstream_proxy_url`,HTTP 采集与浏览器过盾共用;经 `proxyurl.Parse` 校验 scheme 白名单并把 `socks5`→`socks5h` 防 DNS 泄漏,`socks://` 作 `socks5://` 别名见 `normalizeProxyAlias`)+ `cf_clearance` cookie + 配套 `cf_user_agent`(必须与 clearance 配套)+ 超时(单接口 15s / 整次快照 60s)。
**错误分类**(驱动续期/过盾):
- 403/503 + CF 特征(`cf-mitigated=challenge` 或 `cf-ray`+challenge 关键词) → `errUpstreamCFChallenge`
- 401/403 其他 → `errUpstreamAuth`
- 非 2xx → 仅含状态码错误(**不含响应原文**,安全)

## 三、刷新编排与状态机

`upstream_monitor.go` 的 `UpstreamMonitorService` 编排整个刷新生命周期(spec §5.4/§7):

```
RefreshProvider(id, manual)
  ├─ disabled 检查(非 manual 拒绝)
  ├─ ClaimRefresh(now, now-5min)        ← DB 抢占锁(条件 UPDATE),ok=false → ErrUpstreamRefreshInProgress
  ├─ resolveProxy                        ← 从「采集设置」读全局 upstream_proxy_url 进 p.ProxyURL
  ├─ fetchWithRecovery                   ← 见下
  └─ commitSuccess / commitFailure       ← 乐观锁事务提交(WHERE refresh_started_at=ClaimedAt,被抢走则丢弃)
```

### 凭证生命周期 `fetchWithRecovery`(最多重试 1 次)

```
FetchSnapshot
 ├─ errUpstreamCFChallenge → browser 启用? SolveCFClearance → 合并 cf_clearance → 重试
 │                            未启用 → 失败路径
 ├─ errUpstreamAuth → 有账密? Login(纯 HTTP)
 │                     ├─ 成功 → 合并 token → 回写 → 重试
 │                     ├─ 需 Turnstile/CF → browser.AutoLogin → 重试
 │                     └─ 2FA/账密错 → credential_error
 │                    无账密 → credential_error(不重试)
 └─ 其他 → 透传
```

### 状态机(spec §7.2)

| 入态 | 事件 | 出态 | 通知 |
|------|------|------|------|
| any | 成功 | `active`,清零失败计数 | — |
| active | 凭证失效且续期失败 | `credential_error` | 跨入时通知 1 次 |
| active | 连续失败 ≥3 | `unreachable` | 跨入时通知 1 次 |
| active | 失败 <3 | 保持 active | 无 |
| 失败态 | 持续同态 | 不变 | **不重复**(状态未变才发事件) |
| 失败态 | 恢复成功 | active | — |

一次刷新的多条**需通知**变更合并为**一个** `NotifyEvent`(避免轰炸);`balance_recovered` 仅记录不通知。

### 定时 runner

每实例 ticker(1 分钟)→ `ListDue`(到期)→ 并发度 3 → `RefreshProvider(manual=false)`。多实例靠 DB 抢占锁互斥,无 leader 选举;进程退出不优雅停止刷新,靠 5 分钟 stale 锁兜底重抢。

## 四、通知模块(独立、可复用)

- `NotifyDispatcher`:按 `scope`+`enabled`+`events` 过滤渠道,逐个发送;**单渠道失败隔离**(仅记 `last_error`+日志,不影响其他渠道与主流程)。
- `EmailNotifySender`:复用现有 `EmailService.SendEmail` 的 SMTP 能力,内置简洁告警 HTML 模板(条目经 `html.EscapeString` 转义)。
- `WebhookNotifySender`:**SSRF 防护**——复用仓库 `isPrivateIP`(覆盖 loopback/RFC1918/link-local/ULA/CGNAT/0.0.0.0-8 等),DialContext 逐 IP 校验(防 DNS rebinding)、禁止重定向、响应体 64KB 上限;支持 Go `text/template` 自定义 body(`{{.Title}}`/`{{join .Items}}`/`{{json .X}}`),默认标准 JSON。私网默认拒绝,可在「采集设置」放开。

## 五、凭证三层防泄(spec §4.1.1)

1. **响应脱敏**(`RedactUpstreamCredentials`):`password`/`access_token`/`cf_clearance`/`cf_user_agent` 从响应剥离,仅返回布尔状态 + access_token 尾 4 位。
2. **PUT 合并**(`MergeUpstreamCredentials`):敏感键缺省则保留旧值(前端不回显明文也不会误清空);webhook headers 脱敏回写用 `***` 占位,merge 时还原。采集设置的代理 URL 同理——`MaskUpstreamProxyURL` 把密码段脱敏为 `***`,`MergeUpstreamProxyURL` 在回传含 `***` 时保留旧值(空串=清除)。
3. **错误/日志/事件**:`last_error` 经截断且不含响应原文;CF/凭证类事件 `detail` 只存错误类别(`category`),不存凭证值;ent `credentials` 字段标 `Sensitive()`,不进 `String()` 输出。

## 六、余额告警 diff(`upstream_snapshot_diff.go`,纯函数)

跨越式检测:`bal < th && !alerted` → `balance_low`+置位;`bal >= th && alerted` → `balance_recovered`+复位。首次快照(old=nil)只做余额检测。模型价格按键深度 JSON 比较(键序无关)。分组倍率变化/模型增删/分组增删分别产生对应事件。

## 七、浏览器过盾(可选,`upstream_browser.go`)

`ChromedpBrowserSolver` 经 CDP 控制外部 [CloakBrowser](https://github.com/CloakHQ/cloakbrowser)(`cloakhq/cloakbrowser cloakserve`):
- `Enabled` = 采集设置的 CDP 地址非空;未配置 → `SolveCFClearance`/`AutoLogin` 返回 `ErrBrowserDisabled`,刷新走人工兜底。
- `SolveCFClearance`:打开 site_url 轮询等待 `cf_clearance` cookie(60s 上限),返回 cf_clearance + 配套 UA + 保守过期提示。
- `AutoLogin`:填账密 → 等 Turnstile widget → 提交 → 提取登录态(sub2api 读 localStorage `auth_token`;newapi 读 session cookie)。
- `cdpURLFor`:向 CDP URL 注入 `?fingerprint-seed=upstream-<id>`(每 provider 指纹稳定)+ `&proxy=`(QueryEscape,取自「采集设置」全局代理 `upstream_proxy_url`,非 per-provider;经 `browserProxyScheme` 统一为 `socks5://`,Chromium 不识别 `socks5h`)。
- 失败截图:存 `<DataDir>/upstream-diagnostics/<id>/<timestamp>.png`,每 provider 滚动保留最新 5 张;错误信息仅含文件名(不含凭证/页面内容)。诊断文件名严格正则 `^[0-9]{14}\.png$` 防路径穿越。

> dataDir 单一真相源 = `cfg.Pricing.DataDir`(BrowserSolver 写/清、ProviderService 删、Handler 读三处一致)。

## 八、上游消耗采集(fork 新增 phase 1)

> 在余额/价格监控之上叠加的**消耗维度**:把上游"花了多少"采下来,按日×维度落库,供"消耗统计 / 成本对比"。全部新文件,经 `wire_fork.go` 注入。

### 采集管线

- **`UpstreamUsageFetcher` 接口**(`upstream_usage_newapi.go` / `upstream_usage_sub2api.go` 各实现):`FetchUsageDaily(p, sinceDay) → UpstreamUsageResult{CoveredDays, Entries, Partial, Reason}`。适配器内部完成 USD 归一 + 按日×维度聚合;**`CoveredDays` 与 `Entries` 解耦**,区分"查到但当天为空"(应归零)与"查询失败"(应保留)。
  - newapi:翻 `/api/log/self`,`type=2` 计正 / `type=6` 退款计负,`cost=±quota/quota_per_unit`,出 `key`(token_id)/`group`(名)/`model`/`total`。
  - sub2api:列 `/api/v1/keys` → 每密钥 `usage/daily`,出 `key`(api_key_id)+`total`,`cost=actual_cost`;**无 group 维度**。
- **`UpstreamUsageCollector`**(`upstream_usage_collector.go`):单 provider 一轮 = 抢租约 `ClaimCollect` → 算 `sinceDay`(`computeSinceDay`)→ fetch → 三区写库(`buildIncrementWrites` + `WriteDays`)→ 释放租约写回游标。
  - **三区幂等覆盖**:可变窗(最近 `K=2` 天)对覆盖日**先删后插(空=删空,可纠正退款/归零)**;冻结区(已关账旧日)不重查(免被上游清日志误删);补采区(停机漏的新日)append-only;首次**回填 90 天**。
  - 错误处理:fetch 出错 → **不动 rollup**,只记 `last_error`,释放租约。
- **`UpstreamUsageRunner`**(`upstream_usage_runner.go`):独立 ticker(1 分钟),按 `cursor.last_collected_at + provider.refresh_interval_minutes` 判到期后采集;`ProvideUpstreamUsageCollector` 内部 `Start()` 它(仿 `ProvideUpstreamMonitor`)。**手动刷新成功后也追加一次采集**。

### 读取与换算(`upstream_usage_service.go`)

- `usage_summary{today/week/month/total}`:`timezone.StartOf{Day,Week,Month}` 对 `scope_type='total'` 行求和(`SummaryBatch` 一次 SQL 批量,内存聚合避免 N+1);`partial`/`backfilled_from` 取自 `upstream_usage_cursor`。
- breakdown `GET /:id/usage?scope=key|group|model&window=today|week|month|total`:按 `scope_key` 聚合 + 取最新名;`deleted` 比实时 `ListTokens`(密钥,缓存可选)/`latest_snapshot.groups`(分组);**sub2api + scope=group → `supported:false`**。
- **¥ 一律读时换算、只存 USD**(`cost_cny = round(cost_usd / recharge_ratio, 2)`)→ 改 `recharge_ratio` 立即对历史生效。

### 采集并发

`upstream_usage_cursor.collect_started_at` 乐观租约(stale 10min 可重抢,仿 `refresh_started_at` 的 `ClaimRefresh`),ticker / 手动 / 多实例间同 provider 串行,杜绝并发删插竞争,**不复用刷新锁**(隔离两类负载)。
