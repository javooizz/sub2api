# 本地测试与部署

## 一、本地启动(三件套)

一次性配置:在 `deploy/.env` 写入 `POSTGRES_PASSWORD=sub2api`(或留空走默认)。三个终端:

```bash
make dev-deps       # postgres + redis(docker)
make dev-backend    # 后端 go run,首次自动应用 migration 149-151 建表(含消耗 rollup/游标 + recharge_ratio);:8090
make dev-frontend   # 前端 vite HMR;:3000,代理 /api → 8090
```

> `dev-backend` / `dev-frontend` 启动前会**自动清理被占端口**(`SERVER_PORT`/`VITE_DEV_PORT`,缺省 8090/3000),避免旧实例僵死占端口导致"打不开"。

打开 http://localhost:3000 → 管理员登录 → 侧边栏「上游管理」。

## 二、自测本站(无需外部站点)

用 sub2api 自身当上游,可跑通除"过 CF 盾"外的全部功能:

### 1. 添加上游
- 「添加上游」→ 名称随便 / 类型 `sub2api` / 官网地址 `http://localhost:8090`
- 凭证:管理员**邮箱+密码**(若该账号开了 2FA,改用 access_token:浏览器 F12 → Local Storage → 复制 `auth_token` 值填到 Access Token 字段)
- 「测试连接」→ 显示余额 → 保存

### 2. 列表与详情
- 列表看 余额 / 状态 / 分组数 / 模型数 / 最后刷新
- 「详情」抽屉 6 Tab:概览 / 分组价格 / 可用模型 / Token / 关联帐号 / 变更历史

### 3. 余额告警链路
- 「编辑」把**余额阈值设到极高**(如 `99999999`)→ 列表「刷新」
- 详情「变更历史」出现「余额不足」事件 + 「已通知」徽标
- 再次刷新 → 不重复(跨越式);阈值改回低值刷新 → 「余额恢复」事件

### 4. Token / 关联帐号
- Token Tab → 创建 → 一次性弹窗显示明文 key + 有效 API 地址(关闭后不可再看)
- 关联帐号:在账号管理建一个 `type=upstream` 且 `base_url=http://localhost:8090` 的账号 → 详情「关联帐号」能看到它

### 5. 通知渠道(含 SSRF 验证)
- 「通知设置」→ 建 webhook 渠道,URL 填 `http://127.0.0.1:9999/hook` → 测试发送
- 默认私网被拒(400, SSRF);「采集设置」勾选「允许私网 Webhook」后再测 → 变为连接失败(SSRF 已放行)
- 真实收发可用 https://webhook.site 的地址

### 命令行快速验证(可选)

```bash
TOKEN=$(curl -s -X POST localhost:8090/api/v1/auth/login -H 'Content-Type: application/json' \
  -d '{"email":"admin@onebool.com","password":"<密码>"}' | python3 -c 'import sys,json;print(json.load(sys.stdin)["data"]["access_token"])')

# 测试连接(本站,access_token 凭证)
curl -s -X POST localhost:8090/api/v1/admin/upstream-providers/test -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{\"name\":\"self\",\"type\":\"sub2api\",\"site_url\":\"http://localhost:8090\",\"credentials\":{\"access_token\":\"$TOKEN\"}}"
```

## 三、CloakBrowser 过盾(可选,仅上游套 CF 盾时需要)

[CloakBrowser](https://github.com/CloakHQ/cloakbrowser) 是开源 stealth Chromium(C++ 源码级指纹 patch,过 Cloudflare Turnstile),`cloakserve` 模式暴露 CDP 端点 + per-connection fingerprint-seed,正是本模块 `BrowserSolver` 对接的接口。

### 启动 + 配置

```bash
# 启动(首次容器内会下载 ~198MB Chromium)
docker run -d --name sub2api-cloak -p 127.0.0.1:9222:9222 cloakhq/cloakbrowser cloakserve

# 验证 CDP 端点
curl -s http://127.0.0.1:9222/json/version    # 应返回 Browser: Chrome/146...
```

UI「采集设置」CDP 地址填 `ws://127.0.0.1:9222` → 保存。对套盾的上游点「刷新」或「重新登录」,后端会经 chromedp 连 CloakBrowser 打开站点、轮询 `cf_clearance`,拿到后纯 HTTP 带 clearance 采集。

> 容器与后端**出口公网 IP 需一致**(cf_clearance 绑 IP)。Docker Desktop 默认容器经宿主机 NAT 出网,IP 与宿主机相同,可复用。

> **采集代理**:「采集设置」可填一个全局代理(`http(s)://` 或 `socks5://`,也接受 `socks://` 简写),HTTP 采集与 CloakBrowser 过盾**共用**它。过 Cloudflare 通常需要干净的住宅/移动出口 IP——配一个干净代理常能让 HTTP 采集直接通过 CF(login 成功即拿到数据,无需浏览器过盾)。密码在 UI/响应中脱敏为 `***`,留 `***` 不改、清空即移除。

### docker-compose(随仓库)

`deploy/docker-compose.yml` 已内置可选 cloakbrowser 服务(profile `browser`,默认不启用):

```bash
docker compose -f deploy/docker-compose.yml --profile browser up -d
# 容器化后端时,采集设置 CDP 填容器内网地址:ws://cloakbrowser:9222
```

### ⚠️ 已知环境问题:Docker Desktop for Mac 下 TLS 握手失败

在 Docker Desktop for Mac(arm64/amd64 均复现),CloakBrowser 的 patched Chromium 打开 HTTPS 时报:
```
ERROR:ssl_client_socket_impl.cc:926] handshake failed; net_error -100 (CONNECTION_CLOSED)
```
而**容器内系统 `curl` 同站点正常(200)**——说明是 Chromium 对 MTU 敏感、Docker-Mac NAT 下大的 TLS 握手包被丢弃的经典问题(`net_error -100`),非本模块代码问题。本模块的集成链路(chromedp 连接、seed 传递、导航、cf_clearance 轮询、超时优雅返回 400)已验证正常。

**可尝试的规避**(降低容器网络 MTU):
```bash
docker network create --opt com.docker.network.driver.mtu=1400 sub2api-cloaknet
docker run -d --network sub2api-cloaknet --name sub2api-cloak -p 127.0.0.1:9222:9222 cloakhq/cloakbrowser cloakserve
```
Linux 宿主机(生产)通常无此问题。若 MTU 方案无效,可改用其他能过 CF 且暴露标准 CDP 的浏览器(chromedp 连任意标准 CDP 端点即可),或对该上游走人工兜底(手动粘贴 cf_clearance 到凭证)。

## 四、生产部署

1. 五张表(`upstream_providers`/`upstream_change_events`/`notify_channels` + `upstream_usage_daily`/`upstream_usage_cursor`)随 migration 149-151 在后端启动时自动建立;`upstream_providers.recharge_ratio` 列由 151 补加。
2. CloakBrowser 为**可选**:不配 CDP 地址时上游管理仍可用(纯 HTTP + 手动凭证);需过盾时按 §三 起 sidecar 并在采集设置填 CDP 地址。
3. CDP 地址、全局采集代理、webhook 私网开关通过「采集设置」持久化到 `settings` 表(key:`upstream_browser_cdp_url` / `upstream_proxy_url` / `upstream_notify_allow_private_webhook`)。代理 URL 的密码段在 GET 响应脱敏为 `***`。
4. 诊断截图写入 `cfg.Pricing.DataDir/upstream-diagnostics/`,需该目录可写。

## 五、故障排查

| 现象 | 原因 | 处理 |
|------|------|------|
| 前端打不开,`vite proxy error ECONNREFUSED /api/...` | 后端未就绪或端口被僵死进程占着 | 重新 `make dev-backend`(已自动清端口);确认 8090 健康 `curl localhost:8090/api/v1/settings/public` |
| 后端日志 `mkdir /app: read-only file system` | 默认日志文件路径指向容器路径 | **无害**(已降级到 stdout,`LOG_OUTPUT_TO_STDOUT=true` 兜底);如需消除,`deploy/.env` 设 `LOG_OUTPUT_TO_FILE=false` 或指定可写 `LOG_OUTPUT_FILE_PATH` |
| 刷新返回 400「过盾失败: 等待 cf_clearance 超时」 | 未配 CloakBrowser / CloakBrowser 没过盾(见 §三 TLS 问题)/ 出口 IP 被 CF 拦 | **采集设置配一个干净(住宅)代理**常可让 HTTP 采集直接过 CF;否则配 CDP 地址走浏览器过盾;Docker-Mac 试低 MTU 网络 |
| 刷新/测试返回 400「代理地址非法」 | 采集设置的代理 scheme 不在白名单 | 用 `http(s)://` 或 `socks5://`(`socks://` 自动当 socks5),形如 `socks5://user:pass@host:port` |
| 上游状态「凭证异常」 | access_token 失效且无账密续期,或 2FA | 编辑补账密;或重新登录;2FA 账号需用 access_token 凭证 |
| webhook 测试发送被拒(400 SSRF) | 指向私网且未放开 | 采集设置勾选「允许私网 Webhook」 |
| 测试连接失败(本站 sub2api) | 账号开了 2FA,账密 Login 返回 requires_2fa | 改用 access_token 凭证 |
